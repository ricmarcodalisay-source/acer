Master Prompt: Student Learning Application

How to use this: Fill in the [bracketed] sections with your specifics, then paste the whole document into an AI coding assistant (Claude Code, Cursor, etc.) or use it as a spec for a dev team. Sections marked (optional) can be deleted if not relevant.

1. Project Overview

Build a student learning application called [APP NAME] designed to help [TARGET LEARNERS, e.g., "middle school students, ages 11–14"] learn [SUBJECT AREA, e.g., "algebra fundamentals"].

Primary goal: [e.g., "Help students master core algebra concepts through adaptive practice and instant feedback, closing gaps identified by teachers."]

Platform: [Web app / iOS+Android mobile app / Desktop / Cross-platform]

Who uses it:

  • Students — [age range, grade level, tech comfort level]
  • Teachers/Parents (optional) — [do they need a dashboard? assign work? track progress?]
  • Admins (optional) — [school-wide management, classroom rosters]

2. Core Learning Features

  • Content structure: Organize material into [courses → units → lessons → exercises]. Define the specific curriculum/topic list for [SUBJECT].
  • Lesson format: [Video + text / interactive exercises / worked examples / mix]
  • Practice & assessment:
    • Question types: [multiple choice, fill-in-the-blank, short answer, drag-and-drop, code/math input]
    • Instant feedback with explanations (not just right/wrong)
    • Adaptive difficulty — adjust based on performance (describe logic: e.g., "3 correct in a row → increase difficulty")
  • Progress tracking: Visual progress per topic (mastery %, streaks, completed lessons)
  • Spaced repetition / review (optional): Resurface previously learned material at increasing intervals
  • Hints & scaffolding: Provide graduated hints before showing the full answer

3. Motivation & Engagement (optional but recommended for K-12)

  • Gamification: points, badges, streaks, levels
  • Progress visualization: growth charts, mastery maps
  • Light social features: leaderboards, friend challenges (only if age-appropriate and privacy-safe)
  • Encouraging, non-punitive feedback tone — celebrate effort, not just correctness

4. Teacher/Parent Dashboard (if applicable)

  • Assign lessons or practice sets to individuals/groups
  • View student progress, time spent, common errors
  • Export reports (CSV/PDF)
  • Flag students who are struggling on a specific concept

5. Technical Requirements

Stack preferences: [e.g., "React + TypeScript frontend, Node.js/Express backend, PostgreSQL database" — or "no preference, recommend one"]

Must-haves:

  • Responsive design (works on tablets/Chromebooks — common in classrooms)
  • Offline support (optional): cache lessons for low-connectivity environments
  • Accessibility: WCAG 2.1 AA compliance (screen readers, keyboard nav, captions, dyslexia-friendly font option)
  • Performance: fast load times on school networks/older devices

Data & accounts:

  • Auth method: [email/password, Google/Clever/ClassLink SSO, teacher-generated codes for young kids]
  • User roles: student / teacher / parent / admin
  • Data model for: users, courses, lessons, questions, attempts, progress, scores

6. Privacy & Compliance (Critical for Student Data)

  • Comply with COPPA (if any users under 13) and FERPA (if used in US schools)
  • Minimize data collection — collect only what's needed for learning functionality
  • No behavioral advertising or third-party data sale
  • Clear parental/school consent flow for account creation
  • Data deletion/export process for compliance requests
  • (If EU users) GDPR-K considerations for minors

7. Design & UX Guidelines

  • Tone: [encouraging and playful for younger kids / clean and focused for teens/adults]
  • Visual style: [describe — e.g., "bright, friendly, illustration-based" or "minimal, distraction-free"]
  • Avoid: cluttered screens, overly complex navigation, anxiety-inducing timers (unless pedagogically intentional)
  • Include empty/loading/error states designed for a student audience (not raw error codes)

8. Success Metrics

Define how you'll know it's working, e.g.:

  • % of students reaching mastery per unit
  • Average session length / return rate (daily/weekly active users)
  • Reduction in time-to-mastery vs. baseline
  • Teacher-reported usefulness (if applicable)

9. Build Plan / Milestones

  1. MVP: [smallest usable version — e.g., one subject, one question type, basic progress tracking, no dashboard]
  2. V2: [add adaptive difficulty, more content, teacher dashboard]
  3. V3: [gamification, offline mode, analytics]

10. Constraints & Non-Negotiables

  • [e.g., "Must work without an app store install (web-based)"]
  • [e.g., "No ads, no data sale, ever"]
  • [e.g., "Must support Spanish and English"]
  • [Budget/timeline constraints, if relevant]
